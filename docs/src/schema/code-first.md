# Code-first schemas

Instead of clicking together Classes and Properties in the browser first, you can
declare your data model **in code**, publish it with one command, and get
type-safe TypeScript bindings back. Your schema lives in your repo, next to the
code that uses it.

This tutorial walks a JavaScript/TypeScript app builder through the whole loop:

1. Define a schema in code.
2. Publish it to your Atomic Server.
3. Generate typed bindings and use them.
4. **Evolve** the schema — add and change Classes and Properties safely.

> You'll need a running [Atomic Server](../atomic-server.md) and an
> [Agent](../agents.md) (the data-browser creates one for you on first run — copy
> its secret from **User Settings**).

## 1. Define a schema

Create a `todo-schema.mjs`. The shape is a small, JSON-Schema-like subset that
maps onto Atomic [Classes](classes.md), [Properties](classes.md), and
[Datatypes](datatypes.md):

```js
// todo-schema.mjs
export default {
  name: 'TodoApp',
  version: '1.0.0',
  classes: {
    todo: {
      type: 'object',
      description: 'A task in a todo list',
      required: ['title'],
      properties: {
        title: { type: 'string', description: 'Task title' },
        done: { type: 'boolean', description: 'Whether the task is complete' },
        dueAt: { type: 'string', format: 'date', description: 'Due date' },
      },
    },
  },
};
```

Each `properties` entry becomes an Atomic **Property**, each class becomes a
**Class**, and the whole thing becomes one **Ontology**. `type`/`format` map to
Atomic datatypes (`string`→string, `boolean`→boolean, `string`+`format: date`→date,
`integer`→integer, `number`→float, …). Anything not in `required` becomes
*recommended*.

## 2. Publish it

The simplest way is `@tomic/cli`. Create a config once:

```bash
npm install -D @tomic/cli
npx ad-generate init           # writes atomic.config.json
```

Set your `serverUrl` and `agentSecret` in `atomic.config.json`, then publish the
schema and generate bindings in one step:

```bash
npx ad-generate schema ./todo-schema.mjs --generate
```

This creates the Ontology, Classes, and Properties on your server and writes
TypeScript bindings into your output folder. Add `--lock` to also write a
committed, self-verifying `TodoApp.schema.lock.json`.

> **Prefer doing it from code?** Use the SDK directly:
>
> ```js
> import { Agent, Store } from '@tomic/lib';
> import todoSchema from './todo-schema.mjs';
>
> const agent = await Agent.fromSecret(process.env.AGENT_SECRET);
> const store = new Store({ serverUrl: 'http://localhost:9883', agent });
> store.setServerConnected(true);
>
> const registered = await store.registerSchema(todoSchema, { save: true });
> console.log(registered.ontology.subject); // did:ad:…
> ```
>
> This is exactly the [`examples/code-first-schema`](https://github.com/atomicdata-dev/atomic-server/tree/develop/examples/code-first-schema)
> app.

## 3. See it and use it

Open the printed Ontology subject in the data-browser — it renders as a normal,
editable Ontology with your `todo` Class and its Properties.

In code, use the generated bindings for full type-safety:

```ts
import { todoApp } from './ontologies/todoApp'; // generated by --generate

const todo = await store.newResource({
  isA: todoApp.classes.todo,
  propVals: {
    [todoApp.properties.title]: 'Buy milk',
    [todoApp.properties.done]: false,
  },
});
await todo.save();
```

See [@tomic/cli](../js-cli.md) for more on the generated bindings.

## 4. Updating your schema

Your schema file is the source of truth. To change the model, **edit the file and
re-run the same publish/generate command** — then commit the regenerated bindings
(and lockfile) like you would a `package-lock.json`. The git diff shows exactly
what changed.

What happens to identity when you change something is the important part, and
it's designed to keep existing data safe:

- **Add a Class or Property** → new resources are created; everything else is
  untouched.
- **Change only a description, label, or translation** → identifiers stay the
  same. Presentation is not part of a Property's identity, so cosmetic edits never
  churn ids or invalidate data.
- **Change a Property's datatype** (or other machine meaning) → this is a
  breaking change, so it produces a **new** Property. The old one stays valid
  forever, so resources created with it keep working. You're never silently
  reinterpreting existing data.

This immutability is strongest with the **content-addressed** flow. Instead of
`registerSchema`, publish with `registerFrozenSchema` (or click **Freeze** on an
Ontology in the data-browser):

```js
const frozen = await store.registerFrozenSchema(todoSchema, { save: true });
// frozen.properties['todo.title'] === "did:ad:frozen:9c22…"  (a content hash)
```

Here every Property/Class identifier is the hash of its machine meaning. So the
*same* definition always yields the *same* id (re-running is idempotent; identical
definitions across apps even dedupe), and any change yields a new id
deterministically. Pin a release with the committed `*.schema.lock.json` and
[`Store.loadSchemaLock(lock)`](../js-lib/store.md) to make the schema resolvable
offline, with no server. A frozen resource is read-only and shows a **❄ Frozen**
badge in the data-browser.

### Versioning across releases

- Bump `version` in your schema and re-publish for a new release; old frozen ids
  remain resolvable, so old data is never orphaned.
- For a stable "latest" handle, keep a normal (editable) Ontology whose
  `classes`/`properties` point at the current frozen ids — `createSchemaPointer`
  builds one, and its commit history is your version log.

## What you keep from the GUI

Code-first does **not** replace the [Ontology editor](../atomicserver/gui.md) or
[table view](../atomicserver/gui/tables.md) —
they still create the same Classes and Properties, and a code-first Ontology
remains fully viewable and editable in the browser. Use whichever fits: define in
code for repeatable, reviewable, version-controlled schemas; use the GUI for quick
exploration.

## Cross-language

A schema's frozen identifiers are a pure function of its content (RFC 8785 JCS +
BLAKE3), so they are reproducible in any language. The Rust SDK
(`atomic_lib::frozen::freeze_schema`) produces byte-for-byte identical ids — so a
schema authored in JavaScript and one authored in Rust converge on the same
resources.
