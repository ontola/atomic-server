# Encrypted vault sync

> **Status:** Proposal.
>
> This is the public/open counterpart to the SaaS-side "Cloud Vault" idea. It
> describes what `atomic_lib` and open Atomic tooling need so a drive can back
> up and catch up through a blind encrypted object store. It does not describe
> hosted billing, quota, object placement, operations, or other closed SaaS
> infrastructure.

## Goal

Add an open encrypted replication format and local runtime support for a cheap
"blind vault" sync target:

```text
local verifier device
  materializes resources, Loro docs, indexes, queries
  encrypts/decrypts vault objects with drive keys
        |
        v
blind object vault
  stores opaque encrypted objects
  cannot query resources
  cannot inspect file bytes or document text
  does not hold drive keys
```

This is not a hosted AtomicServer. A vault stores encrypted replication objects
for backup and device catch-up. Querying and materialization happen locally.

## Non-goals

- Server-side queries over encrypted drives.
- Server-side search, plugins, previews, automations, or AI over drive content.
- Replacing the existing plaintext verifier-to-verifier sync protocol.
- Defining hosted SaaS billing, quota, abuse controls, or object placement.
- Making a blind replica enforce full resource-level ACLs in v1.

## Relationship to Existing Plans

- [`encryption.md`](./encryption.md) covers the larger design space: verifier
  nodes, blind replicas, key grants, and metadata leakage.
- [`atomic-lib-runtime.md`](./atomic-lib-runtime.md) defines `AtomicNode` as the
  right local runtime boundary.
- [`unified-sync.md`](./unified-sync.md) remains the verifier-to-verifier sync
  model. Vault sync is a separate encrypted object mode, not another transport
  for `SYNC_PUSH`.
- [`s3-blob-storage.md`](./s3-blob-storage.md) covers an S3 backend for normal
  server blobs. Vault sync uses object storage too, but stores encrypted
  replication objects, not queryable Atomic state.
- [`loro-source-of-truth.md`](./loro-source-of-truth.md) still applies inside
  verifier nodes. Vault objects should carry encrypted Loro deltas/checkpoints
  generated by verifiers.

## Product Semantics

Vault sync should provide:

- durable encrypted backup;
- restore to a new device with the drive key;
- catch-up when no verifier peer is online;
- cheap storage-heavy sync;
- local querying after restore/import.

Vault sync should not provide:

- an HTTP alias for the drive;
- browser/public serving;
- live low-latency collaboration guarantees;
- server-side query/index features.

## Open Format Boundary

The following must be open and stable enough for independent recovery tooling:

- encrypted object envelope format;
- manifest format;
- key derivation and key wrapping model;
- object ID derivation;
- restore algorithm;
- corruption detection;
- compatibility/versioning rules.

The hosted implementation can be closed, but users must be able to verify the
threat model and recover their data without the hosted service.

## Required `atomic_lib` Work

### 1. Vault object model

Add a module such as:

```text
lib/src/vault/
  mod.rs
  envelope.rs
  manifest.rs
  keys.rs
  export.rs
  import.rs
  object_store.rs
```

Core types:

```rust
pub struct VaultEnvelope {
    pub version: u16,
    pub drive_id: VaultDriveId,
    pub object_id: VaultObjectId,
    pub kind: VaultObjectKind,
    pub key_epoch: u32,
    pub visible_meta: VaultVisibleMeta,
    pub ciphertext: Vec<u8>,
    pub signature: Vec<u8>,
}

pub enum VaultObjectKind {
    CommitChunk,
    LoroDelta,
    LoroCheckpoint,
    BlobChunk,
    TombstoneChunk,
    Manifest,
}
```

The exact visible metadata is a privacy decision. It must be minimal enough to
avoid unnecessary leakage, but sufficient for:

- fetching missing objects;
- detecting corruption;
- ordering or grouping objects during restore;
- compaction/checkpoint coverage;
- quota and sync progress.

### 2. Drive keys and epochs

Add open primitives for per-drive encryption keys:

```rust
pub struct DriveVaultKey {
    pub drive: Subject,
    pub epoch: u32,
    pub key_material: SecretBytes,
}
```

Needed capabilities:

- create a random drive vault key;
- wrap/unwrap it for an agent or device;
- derive subkeys for envelope encryption, blob chunks, and manifests;
- rotate to a new epoch;
- report which epoch each vault object uses.

The agent signing secret should not become the bulk encryption key. It may sign
objects and unwrap drive keys, but bulk object encryption should use per-drive
key material.

### 3. Export from local verifier state

`AtomicNode` or `Db` needs an export path that produces vault objects from a
materialized local drive:

```rust
pub async fn export_vault_delta(
    node: &AtomicNode,
    drive: &Subject,
    since: Option<VaultCursor>,
    key: &DriveVaultKey,
) -> AtomicResult<VaultExportBatch>;
```

The export should include enough information to restore:

- signed commits or commit chunks;
- Loro deltas since the last exported cursor;
- verifier-created Loro checkpoints;
- tombstones;
- file/blob chunks referenced by current or retained state;
- a manifest that points to the batch and current heads.

Do not export only file blobs. A vault that stores file bytes without commits,
Loro state, tombstones, and manifests is not a restorable Atomic drive.

### 4. Import into local verifier state

Add the inverse restore/catch-up path:

```rust
pub async fn import_vault_batch(
    node: &AtomicNode,
    drive: &Subject,
    key: &DriveVaultKey,
    batch: VaultImportBatch,
) -> AtomicResult<VaultImportReport>;
```

Import must:

- verify object signatures and IDs;
- decrypt with the correct drive key epoch;
- reject corrupted or replayed objects;
- import Loro deltas/checkpoints into local state;
- apply tombstones;
- store file blobs locally or in the configured blob backend;
- rebuild materialized resources and indexes locally.

### 5. Manifest and cursor handling

The vault needs a small open manifest format. It should describe the latest
recoverable state without requiring the object store to understand Atomic
resources.

The manifest should answer:

- which drive this backup belongs to;
- which object format version is used;
- which key epochs are referenced;
- latest known heads/checkpoints;
- batch ordering or cursor information;
- optional compaction coverage;
- integrity hashes for object lists.

Concurrent manifest updates need an explicit rule. The simple v1 option is
compare-and-swap on a manifest generation number, with clients retrying by
merging object lists.

### 6. Object-store abstraction for vaults

Do not couple vault sync to one provider. Add an open trait:

```rust
#[async_trait]
pub trait VaultObjectStore {
    async fn put(&self, id: &VaultObjectId, bytes: &[u8]) -> AtomicResult<()>;
    async fn get(&self, id: &VaultObjectId) -> AtomicResult<Option<Vec<u8>>>;
    async fn list(&self, prefix: &VaultPrefix, cursor: Option<&str>)
        -> AtomicResult<VaultListPage>;
    async fn compare_and_swap_manifest(
        &self,
        expected: Option<&VaultManifestEtag>,
        bytes: &[u8],
    ) -> AtomicResult<VaultManifestEtag>;
}
```

Implementations can include:

- local filesystem for tests and self-hosting;
- generic HTTP/presigned URL adapter;
- direct S3-compatible adapter if useful for self-hosters.

Hosted SaaS may use a closed implementation behind the same open protocol.

### 7. Blob chunk encryption

Plaintext BLAKE3 content hashes leak file equality. Vault blobs should use
ciphertext object IDs or keyed hashes.

Open questions:

- chunk size;
- whether chunking is content-defined or fixed-size;
- whether deduplication is per drive only;
- how to avoid cross-user equality side channels;
- how to represent large file restore without buffering all bytes in memory.

Default leaning: deduplicate only within one drive/account boundary for v1.

### 8. Compaction and retention

A blind vault cannot create or validate plaintext compaction. Verifier devices
must create encrypted checkpoints and sign the coverage metadata.

`atomic_lib` should support:

- creating encrypted checkpoints from local verifier state;
- marking older deltas as covered by a checkpoint;
- validating checkpoint coverage during restore;
- exporting retention hints that a hosted vault may use for GC.

The vault may delete covered objects only after a grace period and only when the
open manifest says the checkpoint is sufficient for recovery.

### 9. Local UX/runtime integration

`AtomicNode` should eventually expose high-level operations:

```rust
pub async fn enable_vault_sync(config: VaultSyncConfig) -> AtomicResult<()>;
pub async fn run_vault_backup_once(drive: Subject) -> AtomicResult<VaultSyncReport>;
pub async fn run_vault_restore(config: VaultRestoreConfig) -> AtomicResult<()>;
pub fn subscribe_vault_status(drive: Subject) -> NodeEventStream;
```

Apps should be able to show:

- last backup time;
- pending bytes/objects;
- last restore/catch-up result;
- key availability;
- quota errors surfaced by the remote service.

## Security Notes

The first public spec must state what remains visible to a vault operator:

- account identity;
- object count and sizes;
- upload/download timing;
- approximate drive activity;
- possibly drive pseudonym or hash;
- IP/device metadata at the transport layer.

It must also state what is hidden:

- resource contents;
- file bytes;
- document text;
- property values;
- local query/index state.

Metadata minimization should be explicit, but v1 should not pretend to hide all
traffic analysis.

## Implementation Phases

### Phase 1: Format and local round-trip

- Define `VaultEnvelope`, `VaultManifest`, `DriveVaultKey`, and cursors.
- Implement local filesystem `VaultObjectStore`.
- Export a small drive to encrypted vault objects.
- Restore into a fresh local `AtomicNode`.
- Test resources, Loro docs, file blobs, and tombstones.

### Phase 2: Incremental backup and catch-up

- Add cursors and incremental export.
- Add manifest compare-and-swap.
- Add corruption/replay tests.
- Add restore from checkpoints plus later deltas.

### Phase 3: Object storage adapters

- Add generic HTTP/presigned URL adapter.
- Optionally add direct S3-compatible adapter for self-hosters.
- Add streaming blob chunk upload/download.

### Phase 4: Runtime integration

- Add `AtomicNode` vault sync operations.
- Add status events.
- Wire into desktop/mobile clients.

### Phase 5: Compaction and GC hints

- Add verifier-created encrypted checkpoints.
- Add retention metadata.
- Add safe deletion rules and tests.

## Test Requirements

- Restore a drive from vault objects into an empty local store.
- Incremental export/import converges with the source drive.
- Loro concurrent edits survive backup and restore.
- Destroyed resources do not resurrect.
- File blobs restore without plaintext hash leakage in object IDs.
- Wrong key epoch fails clearly.
- Corrupted ciphertext fails authentication.
- Manifest conflict handling is deterministic.
- Checkpoint plus later deltas restore the same state as full history.

## Open Questions

- What visible metadata is required in `VaultEnvelope` v1?
- Are vault objects signed by agent, device, or both?
- How are drive vault keys created during normal drive creation?
- How are keys recovered on a new device?
- Should vault sync initially support only same-agent drives?
- How do shared drives distribute vault keys to multiple agents?
- What is the minimum checkpoint format that guarantees restore?
- How should a verifier prove that a checkpoint covers prior updates?
- Should the public docs include a command-line recovery tool before hosted
  vault sync ships?
